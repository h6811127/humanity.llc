Encrypted key backups & recovery keys

Survive a closed tab without giving us your keys.

Live on /created/; hub import of .hcbackup.json on homepage

About this feature

How to restore ›

Pick one path — you do not need all of them. See also Restore on another device and ways you can lose access on the help guide.

  • Encrypted backup — .hcbackup.json file plus the passphrase you chose. The file includes your card ID; no scan link needed. Import in the hub or on /created/ Manage.
  • Recovery code — the Base58 code you saved at create plus your scan link or profile ID. The code alone cannot find your card; there is no handle search. Import both in the hub under Import recovery code.
  • Same browser — if you already saved ownership on this device, tap Open controls. Does not transfer to a new phone.
Why it matters ›

Browser-held keys are powerful and fragile. Without backup, losing the create tab means losing revoke and live-control ability - the sticker still scans, but the owner cannot mutate it.

Design decisions ›

Encrypted backup export uses a user-chosen passphrase (client-side crypto; logic mirrored in Worker tests). Optional recovery key pair can sign revoke and live-control responses separately from the owner key - useful for “break glass” without daily use.

Safety · privacy · security ›

Operator never receives plaintext private keys. Backup blobs are useless without the passphrase. Recovery key display is explicit - user must save it; we do not email or cloud-store secrets.

Limits ›

No “forgot password” recovery from humanity.llc. Weak passphrases or lost backups are owner responsibility. Recovery key compromise equals signing power for that card. Common loss paths: closed tab without backup, cleared site data, iOS storage eviction, recovery code saved without scan link, backup file without passphrase — full list on the help guide.

Future directions ›

Passkey-wrapped backups, hardware wallet integration, optional steward-assisted recovery ceremonies - always opt-in, never default key escrow.
