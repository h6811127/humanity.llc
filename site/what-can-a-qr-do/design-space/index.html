Design dimensions

Patterns that stack on any walkthrough.

Time-bound objects

  • Scheduled availability and office hours
  • Auto-expire and grace periods
  • Intentional dormancy until season start

Request & delegate

  • Scan → ask owner (beyond live control)
  • Scan → offer (finder message, vendor “still here”)
  • Temporary capability: unlock until 6pm

Custody & handoff

  • Held by @handle until Friday
  • Tool-library checkout and return
  • Signed transfer ceremony, not reprint-only

Network primitives

  • Quorum unlock and route state machines
  • Trust warnings propagate along graph
  • Federation and season resets without new IDs

Honest offline

  • Last-known-good with stale disclosure
  • Mesh relay of signed blobs; resolver is SoT
  • Maintainer signs offline, syncs later

Governance & succession

  • Who inherits when signer or org ends
  • Disputed claims on one physical thing
  • Live → read-only archival canon

Anti-patterns

  • No scan streaks or identity leaderboards
  • No “who scanned this bench” surveillance
  • No opaque scores; no irrevocable broadcast

Accessibility & dumb phones

  • NFC tap path; audio capsules on scan
  • SMS/voice short code → resolver status
  • Large-print status for low vision

Blind spots to name explicitly

Gaps in the walkthrough list — missing vocabulary, not missing product.

  • 2 · Time underplayedSchedules, expiry, dormancy — not only game route windows
  • 3 · Custody and handoffCheckout, loan, signed transfer — not only revoke + reprint
  • 4 · Degraded connectivityHonest stale cache and offline maintainer sync
  • 5 · Governance gapsSigner loss, disputes, succession, archival canon
  • 6 · Adversarial physical worldSticker swap, decoy nodes, social engineering at scan
  • 7 · AccessibilityCamera-centric hub; NFC, audio, SMS, no-smartphone paths
  • 8 · Institutional operatorsLibraries, museums, schools — multi-staff signing and audit trails
  • 9 · Deferred domainsHealthcare-adjacent, payments, regulated goods — jurisdiction and liability
  • 10 · Anti-patterns unnamedEngagement loops, surveillance beacons, opaque scores on objects

What a live object can do

Object as endpoint — not QR as label.

  • Multi-tenant streamsCity, artist, and game each own a channel on one bench
  • Append-only historyGuestbook and maintenance log; revoke entries, not only wipe-all
  • Derived stateComputed from checks and quorum without hiding signers
  • Sleep / wakeOff-season archival; reject game scans until season start
  • Rate-limit / quotaObject policy: N contributions per day
  • Opt-in subscribersNotify when shelter opens — not a scan graph
  • District anchorNearby objects share a district ID for map-less discovery
  • Self-describe limitsMachine-readable “I prove X, not Y” on scan

Network of live objects

A network is a graph of signed state with conflict rules — not many QRs in a folder.

  • Quorum / thresholdThree nodes agree → bulletin or unlock updates
  • Route state machineTrail open only if upstream nodes are maintained
  • Trust propagationCompromise at one node → warnings along the graph
  • Coverage mapsWhich crisis resources are live vs stale in an area
  • RedundancyTwo stickers, one logical object; revoke one, other resolves
  • FederationObjects on different operators compose in one scan view
  • Season / act resetLore resets without deleting object IDs
  • Load sheddingMesh/cache reduces resolver hits in dense crowds

Canonical network demo: Cedar Rapids city game and combining ideas.

Technology combinations

Same resolver semantics across carriers. Research directions for NFC, mesh, and Humanity node.

  • NFCSame /c/… URL — wristbands, accessibility
  • BLE meshLocal relay of signed blobs; resolver stays source of truth
  • Solar + e-inkDisplay last resolver state without a phone
  • SMS / voiceShort code → status for dumb phones
  • Calendar (iCal)Signed hours feed for menus and shelters
  • Web Push (opt-in)State-change subscription — not scan analytics
  • Passkeys / hardware keysField maintainer signing without browser tab
  • FederationCross-operator object graphs

Avoid by default: blockchain provenance, permanent IPFS “can’t revoke,” opaque ML trust scores.

Open questions

Product

  • Which verb do we optimize scan UX for first?
  • Who pays for object lifetime?
  • Minimum network size before multiplayer wins?
  • When is a live object worse than paper or a phone call?

Trust

  • Revoked sticker still on the bench — civic UX?
  • Bearer ≠ owner at scale
  • Children, schools, public guestbooks
  • Abuse response vs no scan analytics

Technical

  • Staleness contract for cache/mesh
  • Object outlives its operator
  • Stream stacking → unreadable scan soup
  • Latency: menu sold-out vs AED out of service

Organizational

  • Who may install on public infrastructure?
  • Templates vs one generic create flow
  • City game as canonical network demo?

Competitive

  • NFC-only membership incumbents
  • Apple/Google wallet passes
  • “Just use a Google Form QR”

Ethical

  • Operator neutrality under pressure
  • Gentrification markers on street objects
  • Smartphone-required civic infra
