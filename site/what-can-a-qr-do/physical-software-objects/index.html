Physical software objects

Stickers, clothing, signs, patches, and cards stop being static media. They become deployed software endpoints — scan to interact with a live object, not just open a page.

Research, not shipping yet.

Printing is deployment

Software usually lives on servers, apps, and browsers. A printed QR places a small software reference in the world — with status, permissions, revocation, memory, affiliation, and routing that can change after the ink dries.

  • status
  • permissions
  • ownership
  • revocation
  • expiration
  • memory
  • affiliation
  • game logic
  • public instructions

What you can deploy

  • Sticker Active · revoked · claimed · captured · part of a route or game · bench memory node
  • Hoodie patch Event pass · faction badge · rotating pseudonym · verified affiliation
  • Flyer Live campaign node · current instructions · revoked misinformation
  • Card Public credential · membership pass · recovery-backed · revocable identity

100 stickers = 100 objects

Old model: 100 dead links. Humanity model: 100 deployed references — each with its own live state:

  • 001 · active
  • 002 · captured by Blue faction
  • 003 · revoked
  • 004 · unlocks after midnight
  • 005 · bench memory node
  • 006 · verified for tonight’s event

The sticker does not change. The live object behind it does.

The physical world becomes addressable. Addressable → state.

Deployable public objects

Create a live object, print the QR, place it, manage state over time:

  • active
  • revoked
  • disabled
  • updated
  • claimed
  • transferred
  • trusted
  • expired

Not apps. Not IoT hardware. Cheap printed objects with live network status.

A blog publishes information. An app deploys software. A QR object deploys software into reality — a programmable reference with live state that can change after printing. That is the category-defining idea.
