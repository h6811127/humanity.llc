# Project Brief: Humanity Commons Resolver v0.5

## What This Prototype Was Trying To Build

This repository is a prototype of the Humanity Commons resolver: a small Node/Express service that lets someone create a public Humanity Commons profile, receive a QR code, resolve that QR code back to a public profile, and revoke the profile later with a one-time secret.

The intended product is both a contract/API layer and a minimal browser UI. The contract layer defines how profile IDs, handles, manifesto lines, Ed25519 public keys, and QR payloads are validated. The API layer stores profiles in SQLite, serves resolver endpoints under `/.well-known/hc/v0.5`, and exposes static HTML tools at the site root. The frontend creates a local signing keypair in the browser, posts only the public key to the resolver, displays a QR code and profile URL, and warns users to save revocation material.

The system appears to target the Humanity Commons v0.5 technical spec, with repeated references to:

- Profile storage and SQLite schema.
- Well-known resolver API routes.
- QR payloads of the form `hc://resolve/{profile_id}`.
- Plain-text manifesto and reserved-handle rules.
- Revocation secret hashing.
- Offline profile viewing through a service worker.
- Production deployment through systemd and Nginx on either `humanity.llc` or `resolver.humanity.llc`.

## Intended User Flow

1. A user visits `/create.html`.
2. The browser loads `libsodium.js` and `bs58`.
3. The browser generates an Ed25519 keypair locally.
4. The private key is stored only in browser local storage; the public key is sent to the resolver.
5. The API validates the handle, manifesto line, and public key.
6. The API creates a 20-character base58 profile ID, stores the profile in SQLite, and returns:
   - `profile_id`
   - `profile_url`
   - `qr_code_url`
   - one-time `revocation_secret`
7. The frontend shows the QR image, profile URL, and revocation secret backup controls.
8. Scanning or opening the QR payload resolves to `/.well-known/hc/v0.5/profile/{profile_id}`.
9. The profile endpoint returns HTML for browsers and JSON for API clients.
10. The user can later visit `/revoke.html` and submit `profile_id` plus `revocation_secret`.
11. Revoked profiles and QR images return `410 Gone`.

## System Boundaries

The prototype is not a full identity system. It stores public profile records and revocation secrets, but it does not yet perform signing, authentication, account recovery, key rotation, federation, moderation workflows, or real resolver discovery for the `hc://` scheme.

The local private key generated by the frontend is currently not used by the backend. It is preserved in browser storage as groundwork for future signed actions or stronger revocation flows.

## Components

- `server.js`: Express app factory, route mounting, JSON parsing, rate limit, static frontend serving, service worker headers.
- `lib/`: Core contract, validation, persistence helpers, API route handlers, HTML rendering, request logging.
- `frontend/`: Static MVP UI for create, revoke, profile instructions, offline page, shared app bootstrap, PWA manifest, service worker, CSS.
- `schema.sql`: SQLite profile table and indexes.
- `test/`: Node test suite covering implemented contract, API, frontend static assets, logging, rate limiting, revocation, profile resolution, and QR generation.
- `deploy/`: Example systemd, Nginx, and static marketing sync deployment files.
- `.env.example`: Runtime configuration contract for local and production deployments.
